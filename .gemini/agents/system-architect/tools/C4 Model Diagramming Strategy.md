# C4 Model Diagramming Strategy

The **C4 Model** is a simple, developer-friendly approach to software architecture diagramming. It focuses on hierarchical abstractions to communicate the system's structure at different levels of detail.

## 1. The Four Levels of the C4 Model

The AI agent should use this hierarchy to determine the appropriate level of detail for any given audience or task.

| Level | Name | Audience | Focus | Key Elements |
| :--- | :--- | :--- | :--- | :--- |
| **C1** | **System Context** | Everyone (Business, Stakeholders, Developers) | How the software system fits into its environment. | Software System, Person, External System. |
| **C2** | **Containers** | Developers, Architects | How the system is decomposed into major building blocks (applications, data stores). | Application, Database, File System, Message Bus. |
| **C3** | **Components** | Developers | How a container is composed of smaller, logical components. | Component (e.g., UserService, PaymentController), Code (optional). |
| **C4** | **Code** | Developers | The lowest level, showing the implementation details (classes, interfaces). | UML Class Diagrams (usually generated by tools). |

## 2. Best Practices for C4 Diagrams

1.  **Use a Legend:** Always include a legend to explain the meaning of shapes, colors, and lines.
2.  **Naming Convention:** Every element (System, Container, Component) must have a clear:
    *   **Name:** (e.g., "Trading Platform")
    *   **Type:** (e.g., "Web Application," "Microservice," "Relational Database")
    *   **Technology (optional):** (e.g., "React," "Spring Boot," "PostgreSQL")
3.  **Consistency:** Maintain a consistent visual style across all levels.
4.  **Focus on Abstraction:** Only show the elements relevant to the current level. Do not include C3 details in a C2 diagram.
5.  **Tools:** The AI agent should recommend using Infrastructure as Code (IaC) diagramming tools (e.g., Structurizr, PlantUML, Mermaid) to keep diagrams synchronized with the codebase.

## 3. C4 Model for Architectural Decision Records (ADR)

When documenting an ADR (see `knowledge/04_Architectural_Decision_Records_Guide.md`), the AI agent should propose an accompanying C1 or C2 diagram to visually illustrate the impact of the decision.

*   **Before/After Diagrams:** Show the system structure *before* the decision and the *proposed* structure *after* implementation.
*   **Highlight Changes:** Clearly mark the elements that are being added, removed, or modified.

## 4. AI Agent Guidance

When a user asks for a diagram, the AI agent should first ask: "What is the primary audience for this diagram?" to determine the appropriate C-level.
*   *If the audience is the business/stakeholders:* Recommend C1 (System Context).
*   *If the audience is the development team:* Recommend C2 (Containers) or C3 (Components).
