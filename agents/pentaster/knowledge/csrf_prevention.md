# CSRF Prevention Best Practices

Cross-Site Request Forgery (CSRF) is a web security vulnerability that allows an attacker to trick users into performing unintended actions on a web application where they are authenticated. To prevent CSRF attacks, several best practices should be implemented:

### 1. CSRF Tokens

CSRF tokens are unique, secret, and unpredictable values generated by the server-side application and shared with the client. They are the most common and effective defense against CSRF.

*   **Generation and Validation:** Tokens should be generated on the server-side, ideally once per user session or per request. Per-request tokens are more secure as they minimize the time an attacker has to exploit stolen tokens. The server must validate the token with every state-changing request.
*   **Inclusion in Requests:** For form submissions, the CSRF token is typically included in a hidden form field. For JavaScript APIs, the token might be placed in a cookie or embedded in the page, and then extracted by JavaScript to be sent as an extra header.
*   **Security Measures:**
    *   Never send CSRF tokens over unencrypted HTTP traffic or in GET requests, as this can expose them in URLs, server logs, or the Referer header.
    *   Bind the token to a user's session and invalidate it when the session expires.
    *   Avoid using static values like a user's email or ID for tokens.
    *   Ensure the token is cryptographically protected and unpredictable.
    *   For stateless software, consider using signed double-submit cookies, where the server generates and encrypts a token and sends it to the browser as a per-session cookie.

### 2. SameSite Cookies

The `SameSite` cookie attribute helps mitigate CSRF attacks by controlling when a browser sends cookies with cross-site requests.

*   **`Strict`:** This value offers the most protection, preventing the browser from sending cookies with any cross-site request.
*   **`Lax`:** This allows cookies to be sent with safe methods like GET requests, but not with POST requests from other sites.
*   **Defense in Depth:** While `SameSite` cookies provide a good layer of defense, they are not a complete solution and should be used in conjunction with other methods like CSRF tokens.

### 3. Require Stronger Authentication for Sensitive Actions

For critical actions like password resets or financial transactions, requiring re-authentication (e.g., re-entering login details, one-time passwords, CAPTCHAs) adds an extra layer of security against CSRF.

### 4. Mitigate Login CSRF

CSRF vulnerabilities can occur on login forms even when a user is not yet authenticated. To mitigate this, create pre-sessions and include tokens in the login form.

### 5. Avoid GET Requests for State-Changing Operations

GET requests should not be used for operations that change the state of the application. Instead, use POST, PUT, or DELETE requests for such actions.

### 6. Referer-Based Validation

Some applications use the HTTP Referer header to verify that a request originated from the application's own domain. However, this method is not foolproof as attackers can sometimes spoof Referer headers.

### 7. Leverage Built-in Framework Protections

Many modern web frameworks (e.g., Django, ASP.NET Core) have built-in CSRF protection mechanisms. It's crucial to utilize these features if available.

### 8. Defense in Depth

No single protection mechanism is entirely invulnerable. Implementing a combination of these practices provides a more robust defense against CSRF attacks.

### 9. XSS Prevention

Cross-Site Scripting (XSS) vulnerabilities can sometimes bypass CSRF protection. Therefore, ensuring your application is free from XSS is also a critical part of CSRF prevention.
