# Vite Best Practices for Production

Vite is a next-generation frontend tooling that significantly improves the development experience. This document focuses on optimizing Vite for production builds and integration with Bun.

## 1. Production Build Optimization

Vite uses Rollup for its production build, and these practices ensure the smallest, fastest bundle.

| Practice | Description | Rationale |
| :--- | :--- | :--- |
| **Code Splitting** | Vite automatically splits code, but ensure dynamic imports (`import()`) are used for non-critical routes or components. | Reduces the initial load time by only loading code when it's needed. |
| **Asset Hashing** | Ensure output file names contain content hashes (default in Vite). | Enables long-term caching for static assets, improving performance for returning users. |
| **Minification** | Use the default Terser/Esbuild minification. For maximum performance, ensure the `build.minify` option is set correctly. | Reduces the final JavaScript bundle size. |
| **Image Optimization** | Use a Vite plugin (e.g., `vite-plugin-image-optimizer`) to compress and convert images to modern formats (WebP, AVIF) during the build. | Significantly reduces asset size without manual intervention. |
| **Base URL** | Set the `base` configuration option if the application is not deployed at the root of the domain (e.g., `/my-app/`). | Ensures all asset paths are correctly resolved in the production environment. |

## 2. Vite and Bun Integration

Bun can be used as the package manager and test runner for a Vite project.

1.  **Package Management:** Use `bun install` for dependency management. This is significantly faster than npm or yarn.
2.  **Testing:** Use `vitest` (which is often bundled with Vite templates) and run it with `bun test`. `vitest` is built on Vite's infrastructure, ensuring fast test execution.
3.  **CI/CD:** In the CI pipeline (`tools/01_CI_CD_Pipeline_Strategy_for_Bun.md`), ensure the build step uses `bun run build` to leverage Bun's speed for script execution.

## 3. Development Server Security

The Vite development server is *not* intended for production use.

*   **Never** expose the Vite dev server (`vite` or `bun dev`) directly to the public internet. It is unoptimized and can expose internal file paths.
*   For production, always serve the static files generated by `vite build` using a dedicated web server (e.g., Nginx, or the Bun server for the API).

## 4. Environment Variables

*   **Public Variables:** Variables prefixed with `VITE_` are exposed to the client-side code. **NEVER** store sensitive keys (like API secrets) in these variables.
*   **Private Variables:** Store sensitive variables in the backend (Bun API) and inject them via the K8s Secret mechanism (`knowledge/03_Kubernetes_Deployment_Best_Practices.md`). The frontend should only communicate with the backend API, not directly with external services using secret keys.
